本篇笔记所参考的原 blog 以及图片全部来自知乎用户猛猿，本篇笔记在原文基础上结合了少量自己的理解进行了提炼和整理，用于以后参考和复习。

Pipeline 并行

链接： https://zhuanlan.zhihu.com/p/613196255

为什么要这么设计

模型越来越大，可能导致两个问题：

  1. 中间结果占据的空间越来越大（中间结果需要保存以用来做 bp）；
  2. 模型大到无法在单张卡上运行的情况下，需要把模型拆分到不同的卡上运行，如果采用串行（0 号卡算完一个，1 号卡计算，所有卡算法后，最后一张卡计算 bp，然后逐步传回 0 号卡）则 GPU 利用率非常低。

解决办法

  1. 针对第一个问题，可以不保存所有中间结果，仅保留上一张卡传入的输入，其余中间结果丢弃，需要计算的时候再重算一遍（时间换空间）；
  2. 针对第二个问题，把每个 batch 拆分成更小的 mini-batch，每个 mini-batch 计算完就把结果传到下一个卡，然后计算下一个 mini-batch，以提升 GPU 利用率。

数据并行

链接： https://zhuanlan.zhihu.com/p/617133971 ； https://zhuanlan.zhihu.com/p/618865052

核心思路

在每个 GPU 上拷贝一份完整的模型，各自用一份数据计算一份梯度，然后对梯度进行累加来更新整体梯度。

DP

流程

  • 在每块计算 GPU 上拷贝一份完整的模型参数；
  • 把一份数据均匀分给不同的计算 GPU；
  • 每块 GPU 进行 forward 和 backward 计算后，得到一份梯度 G；
  • 每块计算 GPU 将自己计算得到的梯度传给梯度收集 GPU，进行聚合操作；
  • 梯度收集 GPU 聚合完毕后，计算 GPU 获取完整的梯度结果，用于更新模型参数。

DP 存在的问题

  1. 存储开销大：每块 GPU 上都保存了完整的模型，造成冗余；
  2. 通讯开销大：当梯度收集 GPU 与计算 GPU 不在同一台机器上时，传输带宽会成为效率瓶颈。

DDP

核心思路

利用 Ring-AllReduce，把通讯量均分到每个 GPU 卡上，降低因通讯量集中在某几个卡上导致的延迟。

DeepSpeed ZeRO

优化目标

在大模型中，如果每个 GPU 上都复制一份完整模型，会造成大量冗余且很容易导致 GPU 显存不够用。因此 ZeRO 的主要目标是优化显存占用。

显存占用分析

  • 首先从混合精度训练流程出发，分析显存占用主要由哪些部分构成：
  • GPU 必须存储的内容：
    • Optimizer states：Adam 优化算法中的 momentum 和 variance；
    • Gradients：模型梯度；
    • Parameters：模型参数。
  • 考虑到 fp32 占 4 byte，fp16 占 2 byte，存储量与参数量之间的关系：

ZeRO-DP 优化方式

  • 主要优化思路是，每个 GPU 仅保存一部分参数，需要的时候从其他 GPU 同步过来，计算完成后丢弃不需要的部分，以通讯换空间。
  • 具体流程：
    • 每块 GPU 上仅保存部分参数、梯度以及优化器状态；
    • Forward 时，对权重进行一次聚合，得到完整的权重进行 forward 计算，计算完成后，丢弃其余部分的权重，聚合操作的单卡通讯量为 Φ\PhiΦ；
    • 进行 backward 时，对权重进行一次聚合，得到完整的权重进行 backward 计算，计算完成后，丢弃其余部分的权重，聚合操作的单卡通讯量为 Φ\PhiΦ；
    • 做完 backward，计算一份完整的梯度，对梯度进行一次聚合，每个 GPU 仅需保证自己保存的那部分梯度是聚合了所有 GPU 之后的结果，不需要保证所有梯度都是对的，因此该步操作的单卡通讯量为 Φ\PhiΦ 而不是 2Φ2\Phi2Φ；
    • 用每块 GPU 自己维护的梯度及优化器状态更新自己维护的参数。
  • 与每张 GPU 保存所有参数相比，ZeRO 通讯量增加并不明显，仅仅增加了 1.5 倍，但可以大幅降低显存占用。

张量并行方式

在 MLP 过程中对第一个矩阵采用列切割，第二个矩阵采用行切割。这样做的原因是 $GeLU(A+B)\neq GeLU(A)+GeLU(B)$，如果对第一个矩阵采用行切割的话需要在 $GeLU$ 前进行一次同步；而对第一个矩阵采用列切割可以避免掉这次同步操作。而对第一个矩阵采用列切割后，为了保证第二个矩阵乘法尺寸正确，只能对第二个权重矩阵采用行切割。

张量并行+数据并行

两种并行并存时的安排

两种并行并存时，通常在不同机器之间做数据并行，在同一台机器上做张量并行。

原因

主要原因是张量并行需要在每一层进行通讯以保证结果正确，而数据并行里下一层的计算并不依赖上一层聚合后的结果。因此，张量并行对通讯要求相对更高，在同一台机器上性能更好。

序列并行（Megatron SP）

核心思路

把激活值拆分到不同的 GPU 上，用得到的时候再合并起来。

MLP 层

SP 需要优化的目标

在加入 SP 操作之前，单卡需要存储的激活值为：LayerNorm 前后的激活值 X（4bsh4bsh4bsh）；GeLU 前后的激活值会根据 TP 被分到 N 张卡上，不需要另外处理；Dropout mask 需要在每张卡上保存（bshbshbsh）。因此，SP 需要优化的目标就是每张卡上重复保存的 5bsh5bsh5bsh。

SP 的操作流程

  • 对于 LayerNorm 前的 X，按照 sequence 维度切开，因为 LayerNorm 是在每个 token 上面做 norm，所以按照 sequence 维度切开不影响 LayerNorm 的计算。同时，因为输入是按照 sequence 维度切开的，输出天然也是按照 sequence 维度切开的。
  • 进入 TP 操作前需要把数据同步，需要进行一次 all-gather，以保证每个 GPU 上拿到的都是完整的数据。
  • TP 结束后，按照正常流程需要进行一次 all-reduce 以保证每个卡上都有完整的矩阵相加的结果。但在 SP 中每张卡只需要保证自己计算的那部分 sequence 的结果是正确的就可以，不需要所有结果都正确，因此只需要做一次 reduce-scatter。
  • 因为每张卡只保存自己的那部分 sequence，因此 dropout 也可以只在自己保存的那部分sequence 上进行，所以保存的 dropout mask 也可以切开。这样，MLP 中激活值的冗余部分就被均分到了多个 GPU 上。
  • Bp 时，首先需要进行一次 all-gather，让每张卡拿到完整的梯度结果，然后按照 TP 的流程正常计算 bp。
  • 传到 g 的时候需要进行一次 reduce-scatter，保证每张卡拿到自己需要的那部分梯度结果。
  • 这样通讯量与仅采用 tp 相同（2 次 all-gather+2 次 reduce-scatter 与 2 次 all-reduce），但降低了每张卡的存储量。

Attention 层


操作与 MLP 完全类似，可以把 multi-head attention 操作之外的数据分散到多张卡上。

Selective Activation Recomputation

采用 SP 的情况下还是有可能存在显存不够的情况，Megatron 选择把 attention score softmax 相关的激活值删掉，保存其他激活值。

序列并行（DeepSpeed Ulysses）

运作流程

  1. 把输入数据 X=(N，d)X=(N，d)X=(N，d) 沿 sequence 维度切分，作为各自 GPU 的输入，每个 seq_chunk 的尺寸为 (NP,d)(\frac{N}{P},d)(PN​,d)。
  2. 每张卡计算自己维护的 seq_chunk 的 qkv 的值。
  3. 所有卡间进行一次通讯，使得每张卡拿到所有 sequence 的某 1 个 head 的 qkv。
    • 通讯前每张卡上维护的是某个 seq_chunk 所有 head 的 qkv，通讯后变成所有 sequence 某个 head 的 qkv。
  4. 正常执行 attention 计算，每张卡上计算出某个 head 的结果。
  5. 所有卡间再执行一个通讯，使得每张卡上维护的结果变回某个 seq_chunk 的结果（相当于第 3 步的反向操作）。
  6. 每张卡的结果与 WOW_OWO​ 相乘。
  7. 进入 MLP，因为 MLP 不涉及 token 间的相关性，所以各个 seq_chunk 可以独立计算。

通讯量

All2All 的通讯量

All2All 操作的单卡通讯量为 N×dP×P×(P−1)≈NdP\frac{N\times d}{P\times P}\times (P-1)\approx \frac{Nd}{P}P×PN×d​×(P−1)≈PNd​。通讯量少主要是交换前后每个卡都只保存了一部分数据，如果像 megatron 的话，通讯后每张卡上都是完整数据通讯量就比较高。

Forward 过程通讯量

Attention 计算前，qkv 各进行了一次 All2All 通讯，attention 之后又进行了一次 All2All 通讯，共计 4 次 All2All 通讯。

Backward 过程通讯量

Backward 也需要在上述两个节点进行通讯，使得梯度得到转换，因此通讯量也是 4 次 All2All 通讯。

与 megatron 比较

Megatron 的通讯量与 GPU 数是无关的，也就是当序列变长，无法通过增加更多的 GPU 降低通讯量，但 Ulysses 可以。

序列并行（Ring Attention）

思路

采用了类似 [FlashAttention V2](大模型加速 #FlashAttention V2) 中的分块，Q 作为外层循环，K 和 V 作为内层循环。

多 GPU 版本（环状通信）

单 GPU 版本的性质

  • 先计算哪个 KV 块对结果没有影响，对于固定的 Q 来说，只要最终计算了所有的 KV 块，那么不管顺序是怎么样的，最终结果都是一样的；
  • 一个 KV 块参与计算的时候，其他块是没用的，因此，如果 GPU 上保存了完整的 KV 块是有点浪费的。

多 GPU 优化

  • 每次只保存当前计算的 KV 块；
  • 然后和其他 GPU 通过通讯交换 KV 块。

Chunk_size 选择

  • Chunk size 需要保证通讯时间小于等于计算时间，这样通讯时间可以被计算时间覆盖，不会影响计算效率；
  • Attention 的计算量为：4dc24dc^24dc2（加法和乘法各算一次操作，c 为切好的 sequence chunk 的长度）；
  • 通讯量为：4dc4dc4dc（假设是 16bf 的数据，传输至需要传 KV）；
  • 因此可以推出 c 的约束条件为：C≥FBC\ge\frac{F}{B}C≥BF​ ，也就是不能切地太碎，如果分到太多 GPU 上会影响效率（其中，F 表示硬件运算能力 FLOPS，B 表示硬件带宽 Byte/s）。

序列并行（Megatron Context Parallel）

Ring Attention 的缺陷

Ring attention 假设了所有的 query 与所有的 key 之间计算 attention，但实际上大模型里面不是这样的，基本都是和之前的 key 计算 attention。这样的话按照 ring attention 的逻辑去运转越往后算就越会有很多 gpu 是空置的，这样不同 GPU 的计算负载是不均衡的。

均衡负载版 Ring Attention


用了个比较 tricky 的方法，对于 cp_size，把数据分成 cp_size×2cp\_size\times 2cp_size×2 个 chunk，然后把前面和后面的配成一对，一对 chunk 放到一个 gpu 上计算，这样计算负载就是均衡的。

BBPE 说明 BBPE 算法的说明是在知乎用户 Glan格蓝的文章 https://zhuanlan.zhihu.com/p/652520262 的基础上整理的。 需要分词算法的原因 如果以单个 word 作为单元的话，长尾效应非常严重（会有很多低频词）且无法解决 oov 问题（词表之外的新词）。 BPE 步骤 计算初始词表：比如可以用英文 26 个字母加上各种符号以及常见中文字符，作为初始词表； 构建频率统计：统计所有子词单元对（两个连续的子词）在文本中出现的频率； 合并频率最高的子词对：选择出现频率最高的子词对，将它们合并成一个新的子词单元，并更新词汇表； 重复合并步骤：不断重复步骤 2 和步骤 3，直到达到预定的词汇表大小、合并次数，或者直到不再有有意义的合并； 分词：使用最终得到的词汇表对文本进行分词。 对于 BPE 的理解 其实就是不断找到出现频率最高的组合（词汇表中的两个词链接起来），把这个组合加到词汇表里面，直到词汇表的大小到了预期的大小。但是，这样只能做到把高频词加入词表，没办法解决 oov...
