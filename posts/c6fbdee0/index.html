题目： DeepSeek-Coder-V2: Breaking the Barrier of Closed-Source Models in Code Intelligence
链接： 2406.11931
代码： GitHub - deepseek-ai/DeepSeek-Coder-V2: DeepSeek-Coder-V2: Breaking the Barrier of Closed-Source Models in Code Intelligence
单位： DeepSeek-AI

数据收集

数据构成

  • 60%代码，10%数学相关预料，30%自然语言；
  • 其中，自然语言语料从 DeepSeek-v2 数据里采样。

Github 数据收集

  • 收集 2023 年 11 月前的公开库；
  • 采用与 DeepSeek_Coder 相同的数据清洗规则，具体如下：
    1. 删掉每行平均长度超过 100 个字符或者最大长度超过 1000 字符的文件；
    2. 删掉字母字符占比低于 25%的文件；
    3. 除了 XSLT，删掉 “<?xml version=” 出现在前 100 个字符的文件；
    4. 对于 HTML 文件，保留可见文本占比不低于 20%且长度不小于 100 个字符的文件；
    5. 对于 Json 和 Yaml 文件，仅保留字符数在 50 到 5000 之间的文件；
  • 本文没有描述去重规则，只提到说是和 DeepSeek_Coder 相近的去重规则；
  • 最终数据中，代码数据量为 821B tokens，包括 338 种编程语言，文本数据量为 185B tokens，包括 markdown 以及 issue 之类的文本。

网页数据收集

  • 采用 DeepSeek_Math 相同的 pipeline：
    1. 收集 StackOverFlow、Pytorch documentation 或者 StackExchange 上的网页作为初始语料；
    2. 训练 fastText model；
    3. 根据 base URL 把数据分为不同的 domain。统计每个 domain 的数据在第一轮迭代被分为数学或代码相关的比例，超过 10%的就认为这个 domain 是数学或代码相关的。然后，手工对这些 domain 里与数学或代码相关的部分进行标注。链接到这些 URL 但没在数据集里的网页也被加入 seed corpus；
    4. 3 轮之后，收集到 70B 代码相关，221B 数学相关的 token；
  • 为了收集高质量代码，该流程同样被用于 github 上，初始语料来自手工收集的高质量代码，迭代两轮，这样收集到了 94B 数据；
  • 代码总数据量为 1170B tokens（这个算法应该是 821+185+70+94=1170821+185+70+94=1170821+185+70+94=1170，但是我觉得第一步 github 收集的代码和最后收集的高质量代码应该有重合吧，这部分是否需要处理论文里没提）。

数据集质量

与 DeepSeek_Coder 的数据集相比质量有一定提升。

Training

预训练任务

16B 模型采用 next-token-prediction 和 fill-in-middle 两个任务；236B 模型仅采用 next-token-perdiction 任务（这是大力出奇迹的意思么，模型大了就不需要花里胡哨的东西了）。

模型结构

采用了 DeepSeek-v2 同样的模型，论文中提到了训练过程中遇到了训练不稳定的问题，所以把归一化方法换成了传统归一化方法（DeepSeek 用的应该是 RMS Norm，这个传统的归一化指的是 layer norm 么）。

训练超参数

Long Context Extension

  • 采用 YaRN，超参设计为：s=40,α=1,β=32s=40, \alpha=1,\beta=32s=40,α=1,β=32；
  • 训练采用两阶段：
    • Sequence 长度为 32k，batch size 为 1152，训练 1000 个 epoch；
    • Sequence 长度为 128k，batch size 为 288，训练 1000 个 epoch。

Alignment

SFT

  • 从 DeepSeek_Coder 和 DeepSeek_Math 分别收集 2w 及 3w 条数据；
  • DeepSeek-v2 里也抽了一部分数据，以保证模型在 general 任务上的性能；
  • 总数据集包括 300M token；
  • 初始 lr 为 5e−65e^{-6}5e−6；
  • 训练 batch size 为 1M token，总训练数为 1B token。

RL

Prompts

  • 从各个来源收集了与 code 和 math 相关的 prompt；
  • 约 40k 数据。

Reward modeling

  • 对于数学数据，用 gt 作为监督；
  • 对于代码数据，用编译器反馈的结果（代码是否通过了所有的 test case）作为监督，训练一个 reward model，用 reward model 作为训练 rl（而不是直接把编译器反馈的结果作为 rl 的监督），实验里这样的效果更好。
