说明

本篇笔记所参考的原 blog 以及图片全部来自知乎用户猛猿，本篇笔记在原文基础上结合了少量自己的理解进行了提炼和整理，用于以后参考和复习。

FlashAttention V1

核心优化思路

GPU 的存储包括速度更快但存储空间小的 SRAM 以及存储空间大但速度更慢的 HBM。因此 FlashAttention 想要做的就是尽量充分利用 SRAM，并且将可以合并的操作尽量合并，减少从 HBM 读取数据的操作。

Forward

Attention 标准流程

Safe Softmax

为了防止 softmax 过程出现数据上溢，目前通常采用 safe softmax 的计算方式，即对于每一行，先把该行的所有元素减掉该行的最大值，然后再计算 softmax。

分块计算整体流程

  1. 对 query 矩阵和 key 矩阵切块；
  2. 计算切块后的 query 矩阵和 key 矩阵之间的 attention map，attention map 的尺寸为 Br×BcB_r\times B_cBr​×Bc​，也就是只计算了 attention map 中的一小部分，而不是完整的 attention map；
  3. 对 attention map 计算 safe softmax，mask 和 dropout；
  4. 把 attention map 和分块后的 value 矩阵相乘，得到临时的 output（需要注意该 output 并非完整的 attention map 计算出来的，虽然计算后的尺寸正确，但并不等价于正常计算出的 output）。

简单分块后存在的问题

  1. 分块后如何计算正确的 attention score（因为正常来说要在完整的一行上归一化）？
  2. 分块后如何计算正确的 output？
  3. 分块后是如何实现 I/O 优化，解决 memory-bound 的问题的？

读写量比较

标准 attention 的读写量

标准 attention 需要将 Q 和 K 加载到 SRAM，计算 S，将 S 写回 HBM；加载 S，计算 P，将 P 写回 HBM；加载 P 和 V，计算 O，O 写回 HBM。
总的来说复杂度为 O(Nd+N2)O(Nd+N^2)O(Nd+N2)，其中 N 是序列长度，d 是 embedding 维度。

Flash attention 的读写量

  • 在外层循环会加载 K 和 V，因此循环结束加载一次完整的 K、V，复杂度为 O(Nd)O(Nd)O(Nd)；
  • 内层循环会加载 Q 和 O（l、m 体量较小计算复杂度时不考虑），每次内层循环都会加载一次完整的 Q 和 O，复杂度为 O(TcNd)O(T_cNd)O(Tc​Nd)；
  • 总的来说复杂度是 O(TcNd)=O(NBcNd)=O(4NdMNd)=O(N2d2M)O(T_cNd)=O(\frac{N}{B_c}Nd)=O(\frac{4Nd}{M}Nd)=O(\frac{N^2d^2}{M})O(Tc​Nd)=O(Bc​N​Nd)=O(M4Nd​Nd)=O(MN2d2​)；
  • 这里 M 表示的是 SRAM 的尺寸，只要满足 d2M\frac{d^2}{M}Md2​ 远小于 1，flash attention 就能成功优化读写量。

个人总结

Flash attention 的核心思路是希望通过优化 SRAM 和 HBM 之间的读写交互来提升 attention 的计算效率。具体做法是通过分块计算使得计算过程中 attention map 的读写被省略掉，这样优化掉了一部分读写量，从而加速计算。

FlashAttention V2

核心思路

交换内外循环的顺序，把 Q 放到外循环，K 和 V 放到内循环，这样可以减少 O 的读写。

Backward 流程

  • Backward 过程里还是把 KV 放在外循环，Q 放在内循环。因为 backward 过程中 dK、dV、dQdK、dV、dQdK、dV、dQ 都需要计算和保存，KV 放在外循环，只有 dQdQdQ 需要反复读写，否则的话 dKdKdK 和 dVdVdV 都需要反复读写。

V2 在并行上的优化

  • V1 是按照 batch_size 和 num_heads 划分 block 的；V2 是按照 batch_size，num_heads 和 num_m_block 来划分的，其中 num_m_block 是按照 query 方向做的切分。这样做可以在 batch_size 和 num_heads 相对较小时，增大并行数量，更充分地利用计算资源。

VLLM

原理

和操作系统的那个虚拟页的原理很像。

要解决的问题

大模型的每个回答长度是不确定的，如果给每个请求都按照最长长度预留 KV cache 的空间，则会有很多空间浪费。

解决方式

  • 设置固定大小的 block，并通过映射表，把每个 request 的 block 映射到不同的物理存储位置；
  • 这样不需要给每个 request 分配固定大小的空间，只要上一个 block 满了之后再分配一个新的就行了。

PagedAttention 在不同解码策略下的 trick

Parallel Sampling

  • 对一个 prompt 生成多个不同的回答；
  • 真题思路是先把不同回答的 prompt 都映射到同样的物理空间，等到生成的时候如果发生了分歧，导致某个 block 的内容不完全相同了，再执行复制拆分到不同的物理空间续写。

Beam Search

  • 对每个 token 生成 top k 个结果，下一时刻再从这 top k 个结果出发，生成下一个 token，再保留 top k 个结果。这导致不同的结果之间可能有大量的前置 token 是重合的。
  • Vllm 会及时释放那些失效的 block，例如在下图中，因为 block 8 失效，所以从 block 8 向左，block 4 和 block 2 也会被释放。

调度

要解决的问题

因为没有给每个 prompt 预留足够的空间，可能存在显存已经存满了但是所有 prompt 都没有生成完的情况。

调度原则

先来的请求先被服务，如果显存满了，停掉后来的请求，把先来的请求生成完再把后来的请求恢复执行。后来请求的 KV Cache 会被临时调度到 CPU 上或者直接丢弃掉等着重算。
