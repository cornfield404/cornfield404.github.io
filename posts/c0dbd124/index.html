【DeepSeek系列（二）】DeepSeek Coder

题目： DeepSeek-Coder: When the Large Language Model Meets Programming - The Rise of Code Intelligence
链接： 2401.14196
代码： GitHub - deepseek-ai/DeepSeek-Coder: DeepSeek Coder: Let the Code Write Itself
单位： DeepSeek-AI；Key Lab of HCST (PKU), MOE; SCS, Peking University

数据

数据收集

训练数据包括 87%的源代码，10%的与代码相关的英文自然语言语料库以及 3%与代码无关的中文自然语言语料库。

其中，英语语料库由 github 的 markdown 文档及 StackExchange 构成，用于帮助模型理解代码相关的概念，并且提升模型在 library 调用以及 bug 修复等任务上的能力。中文语料库由高质量文章构成，用于提升模型理解中文的能力。

数据过滤

  • 排除平均行长度大于 100 或最大行长度大于 1000 的文件；
  • 排除字母字符占比低于 25%的文件；
  • 删除 XSLT 编程语言以外的，“<? Xml version=”出现在前 100 个字符的文件；
  • 对于 HTML 文件，保留 visible text 至少占 20%且不少于 100 个 character 的文件；
  • 对于 JSON 和 YAML 文件，仅保留字符数在 50 到 5000 范围内的文件；
  • 删除与测试集相关的代码文件，具体来说采用了测试数据中的 10-gram string 与 3～9 个字符的短字符串过滤数据，匹配上的就删掉文件。

文件间的引用

  • 之前的工作大多在文件级的代码上进行训练，这会导致模型忽略同一项目内文件之间的依赖；
  • DeepSeek 首先解析了文件之间的依赖关系，按照依赖由少到多的顺序排列文件，这样可以保证每个文件所依赖的其他文件在输入序列中位于该文件之前。

Repo 级别去重

DeepSeek 采用 repo 级别的去重，而非文件级别的去重，因为文件级别的去重可能会滤掉 repo 中的某些文件，破坏 repo 的完整性。比如 repo A 和 B 都包含一个文件 c，如果按照文件级别执行去重就可能导致有一个库中的文件 c 被删掉，从而导致该库依赖该文件 c 的文件无法完整找到依赖。

训练

训练任务

  • Next token prediction；
  • Fill in the middle：
    • 主要用于解决代码中常见的知道代码其他部分，需要填其中一块空缺的任务，比如：给定了代码的其他逻辑以及某个函数的功能，需要大模型生成该函数的代码；
    • 方法是随机将文本分成 3 份，并按照 PSM（Prefix-Suffix-Middle）或者 SPM (Suffix-Prefix-Middle) 的顺序排列，这样模型在预测 Middle 的时候就可以看到文本的其他部分；
    • PSM 数据采用的格式：<∣fim_start∣>fpre<∣fim_hole∣>fsuf<∣fim_end∣>fmiddle<∣eostoken∣><|fim\_start|>f_{pre}<|fim\_hole|>f_{suf}<|fim\_end|>f_{middle}<|eos_token|><∣fim_start∣>fpre​<∣fim_hole∣>fsuf​<∣fim_end∣>fmiddle​<∣eost​oken∣>
  • 实验中发现如果 FIM 数据比例过高会影响代码生成能力，因此最终选择了将 50%的数据用 PSM 方式处理。

Tokenizer

在训练语料库的一个子集上用 BPE 构建 vocabulary，vocabulary 的大小为 32000。

训练设置

大多数与 DeepSeek_LLM 里提到的差不多，新提到了 NVLink、NVSwitch 及 InfiniBand。

Long Context

在 RoPE 上采用线性缩放（scaling factor 为 4）并调整了 base frequency 至 100000 后，在 sequence 长度为 16K 的情况下训练了 1000 步。

SFT

  • 利用 Alpaca Instruction format 构建数据集；
  • 为了区分对话里的每一轮，用 <∣EOT∣><|EOT|><∣EOT∣> 作为标识符；
  • 总训练量为 2B token，batch size 为 4M token，初始 lr 为 1e-5，采用 cosine schedule，100 步 warm up。

个人总结

整体是在 DeepSeek LLM 结构的基础上训练的专用于编程的模型，整体结构上没有什么特别特殊的地方。值得学习的两点：

  • 即使是在专用模型上也可以包含通用数据以获得基础的语言理解能力，这个其实也是 LLM 的优势，可以把性质完全不同的数据全部用于训练，提升模型在不同任务上的性能；
  • FIM 的任务形式设计比较巧妙，通过把前后文本放在前面，中间文本放在后面的方式，使得模型在预测中间文本时能看到前后的全部信息，同时利用特殊 token 标记应该插入的位置。通过这种方式，模型在结构上与 next token prediction 任务完全相同。
