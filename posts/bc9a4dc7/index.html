题目： DeepSeek LLM Scaling Open-Source Language Models with Longtermism
链接： 2401.02954
代码： GitHub - deepseek-ai/DeepSeek-LLM: DeepSeek LLM: Let there be answers
单位： DeepSeek-AI

Pre-Training

Data

数据处理主要包含去重、过滤以及 remix 三个步骤。生成的 vocabulary 中的 token 数为 100015，其中有 15 个 special token。

Architecture

Pre-norm structure、RMSNorm function、SwiGLU、RoPE、67B 模型采用了 Grouped-Query Attention

Infrastructures

Megatron、flash attention、ZeRO-1
模型采用 bf16，但是 accumulate gradients 采用 fp32。

Scaling Laws

当计算能力提升时，应该增大模型规模还是增大数据量，以达到最好的模型效果是一个需要研究的问题。

Scaling Laws for Hyperparameters


随着计算能力的提升，batch size 的最佳值增加，learning rate 的最佳值降低。同时最佳值的取值范围都比较宽。

Estimating Optimal Model and Data Scaling

本文采用的复杂度计算公式为：M=72nlayerdmodel2+12nlayerdmodellseqM=72n_{layer}d_{model}^2+12n_{layer}d_{model}l_{seq}M=72nlayer​dmodel2​+12nlayer​dmodel​lseq​。根据实验得到了模型规模及数据规模随计算总量提升的公式，并给出了不同复杂度下的预估 loss。

Scaling Laws with Different Data

当数据集质量增高时，最优分配策略会倾向于在模型上分配更多资源（即增大模型规模）。

Pipeline 并行 链接： https://zhuanlan.zhihu.com/p/613196255 为什么要这么设计 模型越来越大，可能导致两个问题： 中间结果占据的空间越来越大； 模型大到无法在单张卡上运行的情况下，需要把模型拆分到不同的卡上运行，如果采用串行（0 号卡算法一个，1 号卡计算，所有卡算法后，最后一张卡计算 bp，然后逐步传回 0 号卡）则 GPU 利用率非常低。 解决办法 针对第一个问题，可以不保存所有中间结果，仅保留上一张卡传入的输入，其余中间结果丢弃，需要计算的时候再重算一遍（时间换空间）； 针对第二个问题，把每个 batch 拆分成更小的 mini-batch，每个 mini-batch 计算完就把结果传到下一个卡，然后计算下一个 mini-batch，以提升 GPU 利用率。
