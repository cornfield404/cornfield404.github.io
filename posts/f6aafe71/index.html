模型结构

MLA (Multi-Head Latent Attention)

之前方法的问题

  • 传统的 multi-head attention KV cache 需要的空间过大（2nhdhl2n_hd_hl2nh​dh​l），会限制推理时的性能；
  • Multi-query attention 和 grouped-query attention 能缩小 KV cache，但是效果不如 MHA。

Low-Rank Compression

  • MLA 的核心就在与压缩 k、v 的维度以压缩 KC cache，具体做法是先对 h 降维，保存降维后的结果，需要具体计算的时候再升维，恢复到原来的维度。如下面的公式所示，MLA 只需要保存 ctKVc_t^{KV}ctKV​，每个 token 需要的空间是 dcld_cldc​l：

ctKV=WDKVhtktC=WUKctKVvtC=WUVctKV\begin{align} c_t^{KV}&=W^{DKV}h_t \\ k_t^C&=W^{UK}c_t^{KV} \\ v_t^C&=W^{UV}c_t^{KV} \end{align} ctKV​ktC​vtC​​=WDKVht​=WUKctKV​=WUVctKV​​​

  • 经过调整后，在 softmax 阶段：qTk=(WQh)T⋅WUKctKV=hT((WQ)TWUK)ctKVq^Tk=(W^Qh)^T\cdot W^{UK}c_t^{KV}=h^T((W^Q)^TW^{UK})c_t^{KV}qTk=(WQh)T⋅WUKctKV​=hT((WQ)TWUK)ctKV​，因此可以提前计算好 (WQ)TWUK(W^Q)^TW^{UK}(WQ)TWUK，再 inference 的时候就不需要花费额外的时间计算升维；
  • 在论文里，按照 deepseek 的说法 WUVW^{UV}WUV 和 WOW^OWO 也可以合并，但是我没搞懂怎么合并的。网上找了两种解释，一种是苏神的推导，里面所有权重都乘在右边，一种是另一个博主的推导，但是我感觉两种推导的公式都不是常规写法；
  • 论文里还提到可以对 query 降维，虽然这样不会减小 KV cache：

ctQ=WDQhtqtC=WUQctQ\begin{align} c_t^Q&=W^{DQ}h_t \\ q_t^C&=W^{UQ}c_t^Q \end{align} ctQ​qtC​​=WDQht​=WUQctQ​​​

Decoupled Rotary Position Embedding

  • 直接采用常规 RoPE 的问题：通常来说应该对 key 和 query 进行 position embedding，但是如果对 key 进行 position embedding 的话，公式就会变成 qTRoPE(k)=(WQh)T⋅RoPE(WUKctKV)q^TRoPE(k)=(W^Qh)^T\cdot RoPE(W^{UK}c_t^{KV})qTRoPE(k)=(WQh)T⋅RoPE(WUKctKV​)，这样 WQW^QWQ 和 WUKW^{UK}WUK 就没法合并了；
  • 为了解决该问题，MLA 的解决方式是，单独拆出来一部分 feature 用于携带 position embedding 信息。需要注意的是对于 query 来说，每个 head 的 position embedding 是不同的，但是对于 key 来说是相同的（为了减少 KV cache 需要保存的信息，position embedding 这部分某种程度上像个 MQA）：

  • ktRk_t^RktR​ 同样需要被存到 KV cache 里，所以最终每个 token 需要保存的数据量是：(dc+dhR)l(d_c+d_h^R)l(dc​+dhR​)l。

DeepSeekMoE

Key Idea

  1. 划分更细粒度的 expert，这样可以让每个 expert 专用度更高；
  2. 分出一些 shared expert，提取共同的信息，减少每个 expert 之间的冗余。

如上述公式，shared expert 每个都起作用，其余 expert 会用一个 gate 控制，每次选激活程度最高的 KrK_rKr​ 个 expert 生效。

Device-Limited Routing

如果每次激活的 device 太多，会增加通信成本，尤其是 DeepSeek 采用了更细粒度的 expert 划分，这个问题会更严重。因此在 DeepSeek 里限制了每次激活的 expert 最多分布在 M 个 device 上。具体操作上，对每个 token，首先选 M 个激活分数最高的 device，然后在这 M 个 device 上选出 KrK_rKr​ 个 expert。在实验里发现，当 M≥3M\ge 3M≥3 效果就还可以。

参考该 blog，communication balance loss 希望每个 device 接收的 token 尽量少，这样可以降低 device 间的通信。

Token-Dropping Strategy

计算每个 device 的平均负载，对于每个 device 分到的 token 排序，当负载超过平均负载时直接丢掉那些分数最低的 token。训练数据里有 10%的 sequence 不丢弃 token，这样可以让模型在 inference 时能适应丢弃/不丢弃 token 两种方式。

预训练

参数设置

模型超参数

  • 60 层 transformer 层，hidden dimension 5120；
  • 所有可学习的参数以标准差 0.006 随机初始化；
  • MLA 的 head 数为 128，每个 head 的维度是 128，压缩之后的 dcd_cdc​ 是 512，query 压缩之后的维度是 1536，dhRd_h^RdhR​ 是 64；
  • FFN 第一层没用 MoE，其他层都用了 MoE；
  • MoE 有两个 shared experts 和 160 个 routed experts，expert 的 hidden dimension 是 1536，每个 token 会激活 6 个 routed experts（加上 shared experts 一共 8 个）；
  • Compressed latent vectors 和 intermediate hidden states of routed experts 之后加入了 RMS Norm 以稳定训练；
  • DeepSeek-v2 总共有 236B 参数，但是每个 token 只会激活 21B。

训练超参数

  • Optimizer：AdamW，β1=0.9,β2=0.95\beta_1=0.9,\beta_2=0.95β1​=0.9,β2​=0.95，weight_decay=0.1；
  • Learning rate 采用 multi step learning rate，2000 step warm up，训练完成 60%，lr 降为最大值的 31.6%，训练完成 90%，lr 降为最大值的 10%，最大 lr 为 2.4×10−42.4\times 10^{-4}2.4×10−4；
  • Gradient clip 设为 1.0；
  • 在前 225B token 的训练过程中把 batch size 从 2304 逐步增加到 9216；
  • Maximum sequence length 4K；
  • 每一层的 routed experts 被均分在 8 个 device 上；
  • Device-Limited Routing 限定的设备数是 3；
  • α1=0.003,α2=0.05,α3=0.02\alpha_1=0.003,\alpha_2=0.05,\alpha_3=0.02α1​=0.003,α2​=0.05,α3​=0.02；
  • 训练中采用了 Token-Dropping Strategy 以加速训练，测试中没用。

Infrastructures

  • 16 路 zero-bubble pipeline parallelism，8 路 expert parallelism，ZeRO-1 data parallelism；
  • 不需要 tensor parallelism；
  • FlashAttention-2。

Long Context Extension

  • 采用 YaRN，把 sequence 长度从 4K 增加到 128K；
  • 在 sequence 长度 32K，batch size 576 训练 1000 step（在 32K 长度下训练完在 128K 就有还可以的表现，不需要在 128K 训练）。

SFT
